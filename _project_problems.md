# Решение проблем в проекте

## Тема сайта
**Проблема**: свойство `theme` работает только при локальном тестировании
 (на GitHub Pages использует устаревшую версию темы), свойство `remote_theme`
 работает только при публикации сайта на GitHub Pages (при локальном
 тестировании вызывает ошибку).

**Решение**: простейшим решением (без копирования файлов) мне показалось 
что-то вроде предобработчика C++. После недолгих поисков, я нашёл
инструмент `gpp` ("general pre-processor" - общий предобработчик) - предобработчик с поддержкой любых текстовых файлов.
Это позволяет использовать директиву `#ifdef` на шаблоне файла для 
условного включения его части:
```cpp
#ifdef LOCALTEST
theme: minima
#else
remote_theme: "jekyll/minima"
#endif
```
Сценарий `_make_config.sh` запускает `gpp` с параметрами, зависящими 
от своих. Например, параметр `local` задаёт флаг `-DLOCALTEST`, т.е.
в итоговый файл войдёт часть `#ifdef LOCALTEST..#else`, а `#else..#endif` - нет.

## Автовыбор темы
**Проблема**: сценарий, упомянутый выше, не выполняется на компьютере, 
выполняющем сборку сайта для GitHub Pages.

**Решение**: мне впервые потребовалось вносить какие-то изменения в 
задачу GitHub Actions, поэтому для начала я поискал, как это можно
сделать.

На выполняющем компьютере установлена система Ubuntu Linux,
поэтому `gpp`, требуемый сценарию, устанавливается с `apt`:
```sh
sudo apt-get update
sudo apt-get install -y gpp
```
Далее, нужно задать флаг "исполняемый" у сценария, так как на
системах Linux по умолчанию файлы выдают ошибку при попытке
исполнения:
```sh
chmod +x _make_config.sh
```
И, наконец, выполнить сценарий:
```sh
sh _make_config.sh
```

## Персонализация сайта
**Проблема**: создание нового набора правил CSS не имеет эффекта, т.к. 
другой набор применяется после него.

**Решение**: оказалось, что точность селектора (selector specificity) 
влияет на порядок применения. Например, такой селектор:
```css
body .site-header
```
Применяется после следующего: 
```css
.site-header
```
Это происходит потому, что `.site-header` обозначает любой элемент 
с HTML-классом `site-header`, а `body .site-header` - элемент с таким
же классом, но внутри элемента типа `body`, что является более точным
селектором.

## Страница о составе команды
**Проблема**: как можно генерировать страницу, состоящую из 
повторяющихся элементов, без редактирования файлов
HTML?

**Решение**: платформа Jekyll, на которой сделан 
сайт, поддерживает язык программирования Liquid.
При помощи него, считывается `_data/members.yml`:
```
{% for person in site.data.members %}
```

Он содержит краткую информацию об участниках команды:
```yaml
- name: Имя Фамилия
  roles:
    - Список отделов, в которых состоит участник
  photo: Фото участника
...
```

Для каждого участника создаётся контейнер `div`, содержащий
фото, имя и роли в команде. Роли отображаются в виде иконки
с Google Material Icons и названия. Эти данные тоже заполняются
с использованием Liquid:
```html
<img src="{{person.photo}}" />
<p>{{person.name}}</p>
```

Соотнести иконку с названием роли позволяет файл `_data/icons.yml`.
Его ключ `roles` представляет собой словарь перевода названия роли в название
её иконки:

```yaml
Сборка: build_circle
Программирование: code
...
```
Название иконки становится текстом особового `span`:
```html
<span class="material-symbols-outlined" style="vertical-align: bottom;" >{{site.data.icons.roles[role]}}</span>
```

## Шрифт иконок Google Material Icons
**Проблема**: указание более чем четырёх названий иконок в
параметрах CSS-файла Material Icons не позволяет правильно
загрузить его.

**Решение**: как оказалось, если не указывать список нужных
иконок, то успешно загрузятся все, что также упрощает
ссылку на CSS-файл.
